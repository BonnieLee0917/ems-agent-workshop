- **Jingxia Xing**: "harness engineering 没有 hook, harness 不起来，那就是 prompt engineering"
- **Michael Fei**: "hook 本质上是代码来 enforce 确定性，这就回到你昨天发的那个 blog 的 topic 了"

> 📷 Jingxia Xing 发了一张截图（内容未知，请到群里查看原图）

**🧠 解读 & 应用建议:**
**Harness Engineering** 是让 Agent 能长时间稳定运行的工程方法。不是靠 prompt 去"说服"模型，而是用代码中的 hook 来强制确定性行为。比如：Agent 完成一步后必须经过验证才能进入下一步，失败了自动重试，超时了自动恢复。Jingxia 目前能让 Agent 连续跑 8-9 小时，目标是 24 小时不间断。这和传统软件的"守护进程"概念类似。

## 📌 话题三：GPT 5.4 vs Opus — 知识库深度挖掘

**发起人:** Lingyan Zhao
**参与者:** Jingxia Xing

**内容:**

- **Lingyan Zhao**: "目前我用下来对第二大脑的知识做深度链接这件事，做得最好的还是 GPT 5.4"
- **Jingxia Xing**: "怀疑，真的有这么大区别么？"
- **Lingyan Zhao**: "有，我的知识库需要深度挖掘"
- **Lingyan Zhao**: "GPT 5.4 对于一个 concept 能延展得更好"
- **Lingyan Zhao**: "事实上我把知识库丢给 Opus 之后，它写的整理 plan 都没有提出需要抽象出一层 concept，是 GPT 5.4 说的"

**🧠 解读 & 应用建议:**
有意思的实测对比：在知识库整理和概念抽象方面，GPT 5.4 比 Opus 更擅长"深度挖掘"。Opus 擅长执行和编码，但在需要对知识做深层次关联和概念抽象时，GPT 5.4 表现更好。如果你在整理自己的知识体系（比如 ASO 策略库、用户反馈分类），可以试试用 GPT 5.4 做概念提炼，用 Opus 做具体执行。

## 📌 话题四：以数据为中心，不以工具为中心

**发起人:** Xiaolin Quan
**参与者:** Xiaolin Quan

**内容:**

- **Xiaolin Quan**: "我的想法是，最终大家都是分提高 productivity 的，一套完整的知识库是根本，我们搞各种工具要让知识库的数据更快地流动起来：
  - 第一层用工具疯狂地收集数据
  - 第二层自我整理发现
  - 第三层 take action 完成 task，提高效率（这才是最终目的）
  - 同时 feedback 再回到知识库
  - 本质是以数据为中心，不以工具为中心，不再去追求一个完美的工具，追求一个不断成长的 brain"
- 紧接着他也指出："现在的问题是，数据循环太慢了，外面的进来得慢，里面运行的慢，output 不够"

**🧠 解读 & 应用建议:**
这段很有启发。Xiaolin 的框架和你现在让我做的日报就是一个实践：第一层 = 我帮你抓群消息（收集数据），第二层 = 整理分析成日报（自我整理发现），第三层 = 你消化后应用到工作中（take action），feedback 回到你的知识库。可以进一步优化"数据循环"的速度，比如日报里直接标注 action items。

## 📌 话题六：Weipeng Li (Theo Wang) 入群作业 🔥

**发起人:** Weipeng Li (Theo Wang)

**内容:**

Theo Wang 提交了入群申请的"命题小作文"（回答"过去三个月最震惊的 Agent 相关体验"）：

**最震惊的：** Agent 工程的成熟度已经到了一个临界点。
- Harness engineering 让 Agent 不再是一次性对话，而是可被调度、监控、重启的长期运行进程
- 记忆系统让 Agent 能跨任务积累经验，第 100 个 case 比第 1 个做得更快更好
- 多 Agent 编排让复杂任务被拆解成流水线
- 最关键的是验证闭环 — Agent 自己知道做没做对，不需要人盯着

**汗毛竖起来的 moment：**
- 即便模型能力从今天开始冻结，光靠工程手段，Agent 已经足以像"数字员工"一样无人值守地独立完成任务
- 人类的主要工作将切换为：规则制定、任务抽象、流程编排

**正在做的事情：**
- 搭了一个 Agent 调度平台，让多个 Claude Agent 并行处理 Edge 浏览器的 bug 修复，全链路无人介入
- **两周自主修复 150+ 个 case，提交合入 90+ PR**
- 追踪 **Lights-out rate** 指标：有多少比例的代码任务是 Agent 完全自主完成的
- 目标：构建软件工程领域的"黑灯工厂"

> 📷 Theo Wang 附了截图展示 agent 调度平台

**🧠 解读 & 应用建议:**
Theo 的实践数据很震撼：两周 150+ bug fix，90+ PR，全部 agent 自主完成。**"Lights-out rate（黑灯率）"** 是一个很好的度量指标。对 PM 来说，如果 agent 能自主修 bug，PM 的角色就真的变成"规则制定 + 任务分配 + 质量审核"。你可以关注这个方向在 Edge Mobile 的落地可能性。
