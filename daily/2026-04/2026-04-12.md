## 🔧 话题一：AI 驱动的测试 & 调试工作流

**发起人**：Xiaolin Quan | **时间**：16:11

> "以前就是自己打bug, 现在我都不找了，直接说你给我写test case, loop起来写，里面带上opc"

Xiaolin 描述了自己测试方式的转变：不再手动找 bug，而是让 AI 写 test case 并 loop 执行，配合 OPC 并行跑。

**Jingxia Xing**（17:13）回应："看来还是我的问题，项目太少了"。暗示 token 消耗量和项目数量正相关，项目越多越能"烧"起来。

🧠 **解读**：这是从"人找 bug"到"AI 找 bug"的范式转变。loop + test case + OPC 的组合本质上是一个自动化 QA 流水线。对 Edge Mobile PM 来说，这个思路可以直接应用到回归测试上：让 AI 生成 test case → loop 执行 → OPC 并行化，大幅提高测试覆盖率。Jingxia 的"项目太少"也说明了一个事实：token 利用率的天花板取决于你有多少有意义的任务可以并行。

## 🔗 话题二：Telegram Bot 接入 Claude Code / Codex

**发起人**：Xiaochen Wu | **时间**：18:08

> "写了个service, 接了 telegram bot，底层就是 `claude -p xxx` / `codex e "xxx"`，带 session id 就可以接着之前的对话继续聊。"

这是回复 Xiaolin Quan 之前问的"你可以在手机上 resume session id?"。Xiaochen 的方案：自己写一个 service 层，通过 Telegram Bot 作为前端，底层调用 Claude Code CLI 或 Codex，用 session id 实现跨设备的对话续接。

🧠 **解读**：这个方案解决了一个普遍痛点：在电脑上开了一个 CC session，离开工位后想在手机上继续。Xiaochen 的做法是"套壳 IM"，把 CLI 能力暴露到 Telegram。这和我们做 WorkAssit（Teams Bot 套壳 Agent）的思路完全一致。核心启示：**Agent 的入口不一定是专用 UI，现有 IM 平台就是最好的分发渠道**。对 Edge Mobile 产品来说，未来浏览器内置的 AI 助手也应该支持类似的 session 续接能力。
