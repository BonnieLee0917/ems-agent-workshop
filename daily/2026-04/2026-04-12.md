## 🔧 话题一：Agent 工作流实战分享

**参与者**：Xiaolin Quan, Jingxia Xing, Xiaochen Wu | **时间**：16:11 - 18:08

三条消息各自独立但都围绕 agent 工作流优化：

**1. Xiaolin Quan — 用 Agent 写 test case + OPC 循环**
> "以前就是自己打bug, 现在我都不找了，直接说你给我写test case, loop起来写，里面带上opc"

从"手动找 bug"进化到"让 agent 写 test case 并循环执行"。OPC 作为循环引擎嵌入 test 流程。

**2. Jingxia Xing — 项目数量是 token 消耗的关键**
> "看来还是我的问题，项目太少了"

Jingxia 反思自己 token 消耗不够高的原因不是效率问题，而是并行项目数不够。项目越多，agent 能同时处理的任务越多，token 自然就烧起来了。

**3. Xiaochen Wu — Telegram Bot 接入 CC/Codex**
> "写了个 service, 接了 telegram bot，底层就是 claude -p xxx / codex e \"xxx\"，带 session id 就可以接着之前的对话继续聊。"

Xiaochen 搭了一个轻量 service 层，把 Claude Code 和 Codex 包装成 Telegram bot。核心亮点是支持 session id resume，可以在手机上接着之前的对话继续。解决了"离开电脑就断了"的痛点。

🧠 **解读**：三条消息代表三个不同层次的工作流优化。Xiaolin 在**任务设计**层面创新（让 agent 自己写 test）。Jingxia 在**资源分配**层面思考（并行项目数决定 token 利用率）。Xiaochen 在**接入方式**层面突破（手机端随时 resume session）。对 PM 来说，Xiaochen 的 Telegram bot 方案值得关注。如果能在移动端随时和 agent 对话并接续 context，PM 的碎片时间利用率会大幅提升。
